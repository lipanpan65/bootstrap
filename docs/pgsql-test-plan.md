## 测试目标

验证以下三个文件的正确性：

| 文件 | 职责 |
|------|------|
| `common/lib.sh` | 公共函数库（日志、颜色、确认、系统检测） |
| `pgsql/backup.sh` | PostgreSQL 数据库备份（参数解析 → 前置检查 → 执行备份 → 清理过期 → 验证） |
| `pgsql/restore.sh` | PostgreSQL 数据库恢复（参数解析 → 前置检查 → 执行恢复 → 恢复后验证） |

重点验证的能力：

1. **参数解析** — CLI 参数能否正确传递给底层 `pg_dump` / `pg_restore` / `psql`
2. **逻辑分支** — 格式映射、互斥校验、保留策略、格式检测等条件逻辑
3. **安全校验** — 数据库名合法性校验（防 SQL 注入）
4. **过期清理** — 基于文件名时间戳的清理逻辑，是否正确删除过期文件、保留有效文件
5. **端到端流程** — 真实数据库场景下备份 → 恢复 → 数据一致性验证

```
测试分层

┌─────────────────────────────────────────────────────┐
│  第二阶段：集成测试（需要真实 PostgreSQL）            │

```

## 第一阶段：单元测试（mock）

### 运行方式

```bash
# 在项目根目录执行
bash pgsql/test_pgsql.sh
```

测试脚本位于 `pgsql/test_pgsql.sh`，执行后自动输出每条断言的 PASS/FAIL，最终汇总结果。

### mock 策略

单元测试**不需要真实的 PostgreSQL 实例**。通过在临时目录创建 mock 可执行文件，并将其路径插入 `PATH` 头部来替代真实命令：

| mock 命令 | 行为 |
|----------|------|
| `pg_dump` | 将接收到的参数写入日志文件，根据 `-f` 参数创建假备份文件（custom 格式创建普通文件，directory 格式创建目录） |
| `pg_restore` | 将参数写入日志；`--list` 模式返回 3 条假对象记录 |
| `psql` | 将参数写入日志；`-lqt` 模式返回假数据库列表；`-Atc` 模式返回假的大小/计数值 |
| `pg_isready` | 始终返回 0（连接成功） |

测试断言通过**读取 mock 日志文件**来验证脚本是否向底层命令传递了正确的参数。

## 第二阶段：集成测试（真实数据库）

### 前置条件

- 可访问的 PostgreSQL 实例（Docker 或本地安装均可）
- 已安装 `pg_dump`、`pg_restore`、`psql`、`pg_isready` 客户端工具
- 连接参数已知（host / port / user / password）

```bash
# 示例：Docker 环境
docker run -d --name pg_test \
    -e POSTGRES_PASSWORD=testpass \
    -p 5432:5432 \
    postgres:16

# 验证连通性
pg_isready -h 127.0.0.1 -p 5432 -U postgres
```

### 集成测试用例清单

#### E2E-1：整库备份 → 恢复 → 数据一致性

```
步骤：
  1. 在源库创建测试表，插入数据
  2. 执行 backup.sh 备份源库
  3. 执行 restore.sh 恢复到新库
  4. 对比源库和目标库的表数量、行数、数据内容

验证点：
  - 备份文件存在且大小 > 0
  - 目标库表数量与源库一致
  - 关键表的行数与源库一致
  - 抽样数据内容一致（SELECT md5 校验）
```

#### E2E-2：四种格式分别验证

```
对每种格式（custom / directory / tar / plain）：
  1. 备份
  2. 验证备份文件格式正确（pg_restore --list 或 head 检查）
  3. 恢复到新库
  4. 验证数据一致

验证点：
  - custom (.dump)：pg_restore --list 能列出对象
  - directory (.dir)：目录下存在 toc.dat
  - tar (.tar)：文件为有效 tar 格式
  - plain (.sql)：文件头包含 "PostgreSQL database dump"
```

#### E2E-3：选择性备份恢复

```
场景 A — 只备份指定表：
  1. backup.sh -d testdb -t users -t orders
  2. 恢复到新库
  3. 验证只有 users 和 orders 表存在

场景 B — 排除指定表：
  1. backup.sh -d testdb -T audit_logs
  2. 恢复到新库
  3. 验证 audit_logs 表不存在，其余表存在

场景 C — 只备份表结构：
  1. backup.sh -d testdb --schema-only
  2. 恢复到新库
  3. 验证表存在但行数为 0

场景 D — 只备份数据：
  1. 在目标库先创建表结构
  2. backup.sh -d testdb --data-only
  3. 恢复到目标库
  4. 验证数据已恢复
```

#### E2E-4：恢复到已有库（--clean / --no-clean）

```
场景 A — --clean（默认）：
  1. 目标库已有旧数据
  2. 恢复后，旧数据被替换

场景 B — --no-clean：
  1. 目标库已有部分表
  2. 恢复后，验证行为（可能因对象已存在而报 warning）
```

#### E2E-5：恢复到不存在的库（自动创建）

```
  1. 确认目标库不存在
  2. 执行 restore.sh（不手动 createdb）
  3. 验证脚本自动创建了数据库并恢复成功
```

#### E2E-6：并行恢复

```
  1. 备份一个包含多张表的库（custom 格式）
  2. 分别用 -j 1 和 -j 4 恢复
  3. 验证数据一致性（不因并行出错）
```

#### E2E-7：过期清理（真实文件）

```
  1. 在备份目录手动创建模拟的过期文件
  2. 执行一次新备份
  3. 验证过期文件被清理，新文件保留
```

#### E2E-8：连接参数与密码

```
  1. 通过 CLI 参数（-H / -p / -U）指定远程连接
  2. 通过环境变量（PGHOST / PGPORT / PGUSER）指定连接
  3. 通过 .pgpass 文件验证免密连接
  4. 验证 CLI 参数优先级高于环境变量
```

## 已知限制

1. **单元测试不验证数据一致性** — mock 命令不会真正执行 SQL，无法校验备份的数据内容是否完整
2. **mock psql -lqt 返回固定列表** — 如果测试用例中使用的数据库名不在 mock 列表中（testdb / mydb / my_app_db），`preflight` 阶段的"数据库是否存在"检查会失败。需要根据测试用例调整 mock
3. **集成测试依赖外部环境** — 需要可用的 PostgreSQL 实例和客户端工具
4. **不测试网络中断等异常场景** — 如备份过程中数据库断连、磁盘写满等极端情况
5. **不测试 PostgreSQL 版本兼容性** — 脚本在不同 PG 大版本（12/13/14/15/16）下行为可能存在差异（如 tar 格式的 `-Z` 支持在 PG15+ 才有）
